# Electron 版产品路线图 / Roadmap

## 目标

这份路线图的目标，不是把功能堆满，而是帮助 `pixel-agents-electron` 从“好看的 AI 可视化项目”稳步走向“真正可长期使用的 Electron 桌面产品”。

路线设计遵循 3 个原则：

- 优先提升真实使用价值，而不是只加装饰
- 优先强化 Electron 独立版差异化，而不是重复插件版能力
- 优先建立可持续扩展的产品骨架，而不是一次性做很重的系统

一句话方向：

**先把它做成一个好用的多 Agent 桌面工作台，再逐步把它做成一个有成长感、有分享感的 AI 团队操作系统。**

---

## 当前阶段判断

当前项目已经具备比较强的基础能力：

- Electron 独立宿主
- 多 Agent 可视化
- 像素办公室场景
- 自定义角色创建
- 布局编辑器
- 终端 / 会话管理
- 本地数据库与状态持久化
- 桌面端打包分发

这意味着项目已经过了“概念验证”阶段。

下一阶段最重要的问题，不再是“能不能跑起来”，而是：

- 用户为什么要持续开着它
- 用户如何真正管理多个 Agent
- 项目如何从演示感进入产品感
- 哪些功能最适合成为 Electron 独立版的招牌亮点

---

## Roadmap 总览

建议把路线分成 4 个阶段：

1. **Phase 1：从可视化走向可管理**
2. **Phase 2：从管理工具走向有成就感的桌面产品**
3. **Phase 3：从静态角色走向有生命感的团队**
4. **Phase 4：从单机工作台走向平台化系统**

---

## Phase 1：从可视化走向可管理

### 目标

让这个产品不只是“看 Agent 在动”，而是让用户真正开始“调度一支团队”。

### 这一阶段最重要

这是最关键的一阶段。

如果这一阶段做对了，产品会从：

- 一个很酷的像素 UI 工程

变成：

- 一个用户愿意日常开着用的 Electron 桌面工作台

### 核心功能

#### 1. 任务墙 / Task Wall

目标：

- 把当前办公室变成任务分发中心
- 给每个 Agent 一个明确任务来源
- 让用户知道团队到底在忙什么

建议最小版本：

- 增加待办 / 进行中 / 完成三列
- 支持手动创建任务卡
- 支持任务指派到指定 Agent
- 支持任务完成后移动到完成列

成功标准：

- 用户可以不依赖外部脑内记忆来管理当前任务流
- 办公室从“观察界面”升级为“工作面板”

#### 2. 岗位卡 / Role Cards

目标：

- 让每个 Agent 具有清晰身份
- 让用户记住“谁擅长做什么”

建议最小版本：

- 预设岗位：产品、前端、后端、测试、设计、审查
- 岗位影响名称标签、默认提示词、头像装饰或工位元素
- 在 Agent 信息面板中展示岗位

成功标准：

- 用户开始把角色当作“队友”而不是“编号”
- 多 Agent 管理变得更容易区分

#### 3. Agent 信息卡

目标：

- 点击角色时，不只是跳终端，而是看到完整上下文

建议最小版本：

- 名字
- 岗位
- 当前状态
- 当前任务
- 工作目录
- 最近一次工具行为
- 最近一轮完成时间

成功标准：

- 用户可以快速理解单个 Agent 当前情况
- 桌面版信息密度比插件版更强

#### 4. 每日战报 / Daily Report

目标：

- 把一段时间的真实工作转成可总结、可分享的结果页

建议最小版本：

- 今日完成任务数
- 最忙 Agent
- 等待时间最多 Agent
- 使用工具次数统计
- 今日新增角色 / 关闭会话数

成功标准：

- 用户愿意定期查看结果
- 产出可被截图分享

### Phase 1 验收标准

如果这一阶段完成，用户应该能做到：

- 创建一个小团队
- 给团队派任务
- 明确看见谁在做什么
- 明确知道今天做完了什么

## Phase 2：从管理工具走向有成就感的桌面产品

### 目标

让办公室开始承载“进展”和“奖励”，增强留存和情绪价值。

### 核心功能

#### 1. 成果具象化 / Outcome Visualization

目标：

- 让真实完成的工作在办公室留下痕迹

可做形式：

- 白板新增完成记录
- 桌上出现文件或报告
- 房间角落出现奖杯/徽章
- 某工位亮灯
- 某区域多出项目装饰

成功标准：

- 用户一眼能看出最近办公室里发生了哪些进展

#### 2. 办公室成长系统

目标：

- 用真实产出驱动成长，而不是用虚拟货币灌水

建议最小版本：

- 完成一定数量任务后解锁装饰
- 稳定运行一定数量 Agent 后解锁新办公区域
- 某岗位累计工作量达到阈值后解锁岗位专属物件

成功标准：

- 用户愿意长期积累
- 办公室看起来会随着使用越来越“有故事”

#### 3. 功能区域 / Functional Zones

目标：

- 让办公室不同区域具备含义，而不只是摆设

建议最小版本：

- 会议区：审查 / 讨论时有特殊演出
- 休息区：长等待状态时短暂停留
- 招募区：创建 Agent 的专属入口
- 项目区：某些任务显示在特定区域

成功标准：

- 办公室开始成为一种“功能布局”，而不只是视觉布局

### Phase 2 验收标准

如果这一阶段完成，用户应该能感受到：

- 真实工作在办公室里留下了成果
- 产品不再只是效率工具，也有成长反馈
- 办公室越来越像自己的团队空间

---

## Phase 3：从静态角色走向有生命感的团队

### 目标

让角色有更强的生命感和记忆点，但不把系统做成重养成游戏。

### 核心功能

#### 1. 子 Agent 跑腿演出

目标：

- 把 sub-agent 变成清晰可见的协作过程

建议最小版本：

- 父 Agent 发起子任务
- 临时角色出现并移动执行
- 返回后汇报结果并消失

成功标准：

- 多 Agent 协作拥有非常直观的视觉表达
- 成为产品展示中的强记忆点

#### 2. 情绪 / 精力反馈

目标：

- 给角色增加轻度情绪反馈，提高代入感

建议最小版本：

- 忙碌、等待、顺利完成、被阻塞几种轻状态
- 只做视觉 / 动作变化，不做复杂惩罚数值

成功标准：

- 用户开始注意角色状态差异
- 办公室变得更有“人在工作”的感觉

#### 3. 协作关系 / Collaboration History

目标：

- 让角色之间的长期协作逐渐被感知

建议最小版本：

- 显示谁和谁经常搭档
- 某角色最常承担哪种任务
- 最近协作历史摘要

成功标准：

- 用户会逐渐形成“团队记忆”
- 角色不再只是随机临时实例

### Phase 3 验收标准

如果这一阶段完成，用户应该会觉得：

- 这些 Agent 像一支团队，而不是一批窗口
- 每个角色逐渐有了自己的“个性”和位置

---

## Phase 4：从单机工作台走向平台化系统

### 目标

为后续更大方向留接口，但不要提前过度实现。

### 可能方向

#### 1. 多项目 / 多办公室系统

- 一个办公室对应一个项目
- 多项目切换
- 不同项目有不同空间主题

#### 2. 更丰富的适配器系统

- 不只支持 Claude Code
- 逐步考虑接入更多 agent runtime

#### 3. 主题与素材生态

- 更多官方办公室主题
- 第三方角色 / 皮肤 / 素材包
- 社区可扩展生态

#### 4. 团队级分享能力

- 战报导出
- 项目快照
- 办公室状态分享图

### Phase 4 验收标准

这一阶段不是短期目标。

它的意义是：

- 让产品未来有平台化空间
- 但不影响前面三阶段先把核心体验打透

---

## 推荐执行顺序

如果只按最稳的产品节奏推进，建议顺序如下：

### 第一优先级

- 任务墙
- 岗位卡
- Agent 信息卡
- 每日战报

### 第二优先级

- 成果具象化
- 办公室成长系统
- 功能区域

### 第三优先级

- 子 Agent 跑腿演出
- 情绪反馈
- 协作关系

### 第四优先级

- 多办公室
- 平台化适配器
- 更强分享能力

---

## 不建议当前优先投入的方向

### 1. 纯装饰型细节过多

例如：

- 只增加随机动作
- 只增加表情
- 只增加小摆件但没有功能含义

这些可以提升可爱度，但不能作为路线核心。

### 2. 重数值养成

例如：

- 复杂金币系统
- 长线资源循环
- 过于游戏化的疲劳惩罚

这会削弱它作为真实工作台的价值。

### 3. 过早多人协作

先把单用户管理多 Agent 的体验做顺，
再考虑多人共用空间或共享团队。

---

## 对你当前项目最有价值的结论

从现在这个 Electron 独立版的发展阶段看，最关键的一步不是继续加“可爱小功能”，而是要补齐下面这条主链路：

**任务分发 → 角色身份 → 工作反馈 → 结果总结**

也就是：

1. 有地方派任务
2. 有明确的人设和职责
3. 有清晰的工作状态与成果反馈
4. 有阶段性总结与分享出口

只要这条链打通，这个项目就会从：

- 一个有趣的像素 AI 工具

升级成：

- 一个真正有产品势能的 Electron 桌面应用
